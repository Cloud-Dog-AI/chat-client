### Server Architecture

This project runs servers controlled via `server_control.sh`:
- **API Server** (port 8083) — REST API for chat operations

```bash
# Docker builds:

```

### Cache File Management

- **NEVER** edit cache files directly (`cache/context.json`, `cache/system_context.md`, etc.)
- Use API endpoints to rebuild/update context
- Cache files are generated by the system, not manually edited

### Context API Endpoints

**Read-only endpoints (for status checks):**
- `GET /health` — server health (public, no auth)
- `GET /context/validate` — context validation (read-only, requires API key)

**Write endpoints (NEVER use for status checks):**
- `POST /context/update?full_update=true` — context rebuild
- `POST /context/rebuild/*` — WRITE operations

Authentication: `X-API-Key` header, from env var

### LLM Configuration

- **Ollama must be remote only**: `https://llm1.cloud-dog.net`
- Approved models: `qwen3:14b`, `granite4:tiny-h`, `gemma3:12b`
- **Preflight check required**: Run `curl -fsS <ollama_base_url>/api/tags` before any Ollama test — if curl fails, test MUST fail with clear error

### External Service Interfaces (Single Entry Points)

| External Service | Single Entry Point | Direct calls forbidden |
|-----------------|-------------------|----------------------|
| LLM | `src/llm/llm_service.py` | No direct Ollama/OpenAI calls |
| Database | Database service layer | No direct SQL from business logic |
| MCP | `src/cloud_dog_chat_client/mcp/client.py` | No direct MCP transport calls |
